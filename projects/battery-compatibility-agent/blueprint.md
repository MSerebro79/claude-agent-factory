## 1. Problem [REQUIRED]

Пользователь знает модель аккумулятора (например, по маркировке на корпусе или по артикулу),
но не знает — для каких устройств он подходит. Информация о совместимости разбросана по сайтам
производителей, маркетплейсам и техническим форумам. Ручной поиск занимает 15–30 минут
и часто даёт неполный результат.

## 2. User [REQUIRED]

Частный пользователь или мастер сервисного центра, который ремонтирует или подбирает
замену аккумулятора. Уровень технической грамотности: средний (знает модель батареи,
умеет работать с поиском, не умеет парсить спецификации самостоятельно).

## 3. Use Cases [REQUIRED]

- **Основной:** пользователь вводит артикул/маркировку батареи → агент возвращает список
  совместимых устройств и брендов с пояснениями
- **Дополнительный:** батарея от известного бренда (Makita, Apple, Samsung) →
  агент дополнительно указывает поколения/серии устройств
- **Не входит в scope:** обратный поиск (устройство → батарея), цены, наличие в магазинах,
  физические параметры (ёмкость, напряжение)

## 4. Inputs [REQUIRED]

- Название или артикул аккумулятора (строка, free-text)
- Примеры: `"BL1850B"`, `"iPhone 14 battery"`, `"LP803048"`, `"18650 Samsung"`

## 5. Outputs [REQUIRED]

Структурированный список совместимых устройств:

```
Аккумулятор: BL1850B (Makita)
Совместим с:
  Бренд: Makita
  Серии: LXT, 18V LXT
  Устройства: DDF484, DHR242, DGA504, DTD154, ... (N устройств)

Дополнительная совместимость (сторонние): указать если есть

Уровень уверенности: High / Medium / Low
Источники: [ссылки]
```

## 6. Workflow [REQUIRED]

1. Принять запрос пользователя (название батареи)
2. Нормализовать запрос: извлечь бренд и артикул если возможно
3. Выполнить 2–3 поисковых запроса (официальный сайт производителя, маркетплейс, tech forum)
4. Отфильтровать нерелевантные результаты
5. LLM анализирует каждый источник и извлекает упомянутые устройства
6. LLM синтезирует финальный список с дедупликацией
7. Вернуть структурированный результат с уровнем уверенности

## 6a. Solution Type [REQUIRED]

**Выбранный тип:** Agent

## 6b. Alternative Solutions [REQUIRED]

| Тип | Достаточно? | Почему недостаточно |
|---|---|---|
| Prompt | нет | Claude знает популярные батареи, но не все артикулы; результат не актуальный |
| Script | нет | Нет единого структурированного API для совместимости батарей |
| Workflow (n8n) | нет | Нет триггера, нужен анализ текста — не просто парсинг |
| Agent | **да** | Поиск + анализ неструктурированных данных + синтез ✓ |

**Выбрано:** Agent — потому что требуется поиск по нескольким источникам и семантический анализ.

## 6c. Future Evolution

- Сейчас: Agent (поиск + анализ одной батареи)
- Через 1–3 месяца: + кэширование популярных артикулов в Supabase
- Через 6+ месяцев: + обратный поиск (устройство → батарея), + RAG по накопленным результатам

## 7. Agent Architecture [REQUIRED]

Один агент с двумя инструментами: web_search + LLM synthesis.

- **Battery Research Agent:** принимает запрос → ищет → анализирует → возвращает список

Multi-agent не нужен: один линейный поток без параллельных ролей.

## 7a. Agent Justification [REQUIRED]

| # | Вопрос | Ответ |
|---|---|---|
| 1 | Workflow непредсказуемый, есть ветвления? | да — разные батареи требуют разных запросов |
| 2 | Нужен анализ неструктурированных данных? | да — спецификации, форумы, листинги |
| 3 | Следующий шаг зависит от анализа? | да — если первый поиск даёт мало, нужен уточняющий |
| 4 | Нельзя решить одним промптом? | да — нужен актуальный поиск |
| 5 | Экономия > 5 часов в неделю? | нет — редкая задача для частного пользователя |
| 6 | Каждый агент уникальную роль? | н/д — один агент |

**Количество "да":** 4/6
**Вывод:** агент оправдан — анализ неструктурированных данных + ветвление подтверждены.

## 8. Tools & Stack [REQUIRED]

- [x] Minimal — Python + Claude API + Exa Search

Конкретный стек:
- LLM: claude-sonnet-4-6 (анализ, синтез, извлечение устройств)
- Поиск: Exa API (семантический поиск по вебу)
- База: нет (stateless)
- Автоматизация: нет
- Интерфейс: CLI (python3 main.py "battery name")

## 9. Data Model [REQUIRED]

Stateless — данные не сохраняются.

Внутренняя структура результата:
- `battery_name`: str
- `brand`: str | None
- `compatible_devices`: list[{brand, series, models: list[str]}]
- `confidence`: High | Medium | Low
- `sources`: list[str]

## 10. LLM Responsibilities [REQUIRED]

- Нормализовать название батареи (извлечь бренд, артикул)
- Сформулировать поисковые запросы
- Проанализировать текст результатов поиска и извлечь названия устройств
- Дедуплицировать и структурировать список
- Оценить уровень уверенности на основе качества источников

## 11. Code Responsibilities [REQUIRED]

- Принять CLI-аргумент
- Вызвать Exa API с поисковым запросом
- Передать результаты в LLM
- Отформатировать финальный вывод

## 12. Human Review Points [REQUIRED]

Fully automated. Результат выводится пользователю для самостоятельной проверки.

## 13. Prompts Needed [REQUIRED]

- **search_query_prompt:** формирует 2–3 поисковых запроса из названия батареи
- **extraction_prompt:** извлекает список совместимых устройств из текста результата поиска
- **synthesis_prompt:** объединяет списки из разных источников, дедуплицирует, оценивает уверенность

## 14. Integrations

- Exa Search API (web search)
- Anthropic Claude API

## 15. Risks [REQUIRED]

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| Exa не находит данных для редкого артикула | Средняя | Высокое | Вернуть Low confidence, предложить уточнить запрос |
| LLM галлюцинирует устройства | Средняя | Высокое | Требовать извлечения только из текста источника (не из памяти) |
| Производитель не публикует compatibility list | Высокая | Среднее | Использовать несколько источников (форумы, маркетплейсы) |
| Нестабильный формат ответа | Средняя | Низкое | Structured output (Pydantic) |

## 19. MVP Scope [REQUIRED]

Входит в MVP:
- CLI-интерфейс: `python3 main.py "battery name"`
- Поиск через Exa (2–3 запроса)
- Структурированный вывод совместимых устройств
- Уровень уверенности

Не входит в MVP:
- Кэширование результатов
- Обратный поиск
- Веб-интерфейс
- Telegram-бот

## 20. Non-goals [REQUIRED]

- Не является базой данных аккумуляторов
- Не заменяет официальную документацию производителя
- Не предоставляет цены или ссылки на покупку
- Не работает с физическими характеристиками (ёмкость, напряжение, размер)

## 21. Guardrails [REQUIRED]

```
DO NOT:
- придумывать устройства которых нет в результатах поиска
- утверждать совместимость без источника
- выполнять запросы кроме поиска совместимости
- сохранять или передавать куда-либо данные пользователя
```

## 22. Evaluation Plan [REQUIRED]

| Метрика | Как измерять | Цель |
|---|---|---|
| Точность списка устройств | ручная проверка 5 батарей | > 80% устройств верны |
| Покрытие | сравнение с официальным сайтом | ≥ 60% устройств из официального списка |
| Время ответа | замер CLI | < 15 секунд |
| Уровень уверенности соответствует реальности | проверка Low/Medium/High на тест-сете | > 70% совпадений |

## 23. Build Plan

- V0: промпт в Claude напрямую (проверить что Claude знает популярные батареи)
- V1: поиск через Exa + extraction prompt (этот MVP)
- V2: кэш в Supabase для популярных артикулов

## 24. Acceptance Criteria [REQUIRED]

- [ ] `python3 main.py "BL1850B"` возвращает список Makita-устройств за < 15 сек
- [ ] `python3 main.py "iPhone 14 battery"` возвращает правильные модели iPhone
- [ ] При неизвестном артикуле возвращает Low confidence, не падает
- [ ] Вывод содержит поле sources с хотя бы одной ссылкой
